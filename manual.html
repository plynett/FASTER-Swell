Technical Information

Users Manual and Technical Basis

FASTER-Swell is a low-computation coastal forecast workflow for estimating maximum wave crest elevation, 2% exceedance runup, overtopping likelihood, and wave-driven flooding potential along California shorelines. The Explorer combines nearshore wave forecasts from the CDIP MOP alongshore model, NOAA tide predictions, and a simplified user-edited beach or structure profile to produce a transect-by-transect operational view.

Primary objective Provide a short-term swell impact forecast that helps interpret flooding, overtopping, berm performance, and pier or wharf exposure.
Hydrodynamic inputs CDIP MOP wave height, peak period, peak direction, prediction depth, and nearest NOAA tide predictions in MLLW.
Geometry inputs User-defined slope, toe elevation, toe distance, crest elevation, crest distance, sea-level anomaly, and local chop adjustment.
Outputs Wave and tide playback, maximum crest, expected runup, conservative runup, upper-bound runup, and overtopping category.

Purpose and Scope

The background documents describe FASTER-Swell as a forecast tool for large wind-wave and swell events, not a tsunami model. The method is intended for event windows on the order of several days and is designed to answer questions such as: How high can wave crests reach near a pier deck? Will a temporary berm or dune likely be overtopped? Is wave-driven flooding likely on a coastal road or backshore area?

The operational concept is to start with wave conditions already predicted by the MOP system near the 10 m depth contour, combine those conditions with NOAA tides and a user-defined shoreline geometry, then evaluate several empirical runup formulations to produce decision-ready water-level estimates.

Operational Data Flow

  1. The Explorer loads statewide MOP transects from transects.kml via the compact web export.
  2. When a transect is selected, the app requests the corresponding CDIP MOP forecast directly from the THREDDS DAP2 ASCII endpoint.
  3. The app resolves the nearest NOAA tide station from the California station list and downloads hourly tide predictions in MLLW.
  4. The Geometry Editor defines a stylized profile using the user inputs. Dune mode returns landward to the toe elevation; structure mode remains elevated landward of crest.
  5. For each forecast time, the app computes equivalent deep-water wave properties, breaker conditions, runup estimates, and summary water levels.
  6. The Explorer displays the wave and tide time series, allows playback through forecast time, and plots the selected-time water levels across the edited profile.

Geometry Representation

Beach face and toe location

zmin = -5 m (internal reference)
xlinear,end = (ztoe - zmin) / β
xflat = xlinear,end + Wtoe

The Explorer builds the seaward profile from a constant average slope β up to the dune or structure toe elevation, then applies a flat toe/berm segment.

Source: current Explorer implementation in web/app.js.

Dune or structure crest shape

Wd = 2 Wtoe-crest
xc = xflat + Wd / 2
A = zcrest - ztoe
σ = Wd / 6
z(x) = ztoe + A exp(-(x - xc)2 / (2σ2))

The dune/structure crest is represented with a Gaussian rise over the specified toe-to-crest distance. In dune mode the landward side returns to the toe elevation; in structure mode the landward side remains at crest elevation.

Source: current Explorer implementation in web/app.js.

Hydrodynamic and Runup Equations

Terminology note

The local project files sometimes refer to the Stockdon formulation as "Stockton." In this manual the published name Stockdon et al. (2006) is used. The formulas below match the project formula sheet and the current code path.

1. Conversion from nearshore prediction point to equivalent deep-water properties

Shallow-water and deep-water wavelengths

Ls = (g Tp2 / 2π) √tanh(4π2d / (Tp2 g))
Lo = g Tp2 / (2π)

These are the nearshore and equivalent deep-water wavelengths used throughout the transformation and runup calculations.

Source: project formulas sheet; implemented in FASTER_MOPS.py and web/app.js.

Group factor, wave angle, shoaling, refraction, deep-water wave height

k = 2π / Ls
ns = 0.5 [1 + 2kd / sinh(2kd)]
angs = Dp - θshore-normal
angd = sin-1[(Lo / Ls) sin(angs)]
Ks = √[nsLs / (0.5Lo)]
Kr = √[cos(angs) / cos(angd)]
Ho = Ks Kr Hs,swell

The documented method uses the forecast mean wave direction relative to shoreline normal. The present Python and Explorer implementation currently set the operational directional term to zero inside the timestep runup calculation, so the live calculation behaves as a non-directional baseline.

Sources: background docx, formulas sheet, O'Reilly et al. (2016), current code.

2. Breaking-wave calculation and maximum crest elevation

CEM-style breaking relation

a = 43.75 [1 - exp(-19β)]
b = 1.56 / [1 + exp(-19.5β)]
Hb = Ho Ks,b Kr,b
dbreaking = Hb / [b - a Hb / Tp2]

The solver iterates on breaking depth until the Coastal Engineering Manual style breaking relation is satisfied.

Source: project formulas sheet, attributed there to the Coastal Engineering Manual breaking-wave-height method; implemented in Python and web code.

Maximum-considered wave crest

zcrest,max = η + Hb MaxRatio CrestRatio
MaxRatio = 1.3
CrestRatio = 0.55

Here η is tide plus user-specified sea-level anomaly. The formula sheet treats 1.3 as a representative ratio between maximum-considered wave height and significant wave height, and 0.55 as the FEMA crest fraction of total wave height.

Source: project formulas sheet; implemented in FASTER_MOPS.py and web/app.js.

3. 2% exceedance runup formulas

Stockdon et al. (2006) total runup

SetupStockdon = 1.1 0.35 β √(LoHo)
SwashStockdon = 1.1 0.5 √[HoLo(0.563β2 + 0.0004)]
R2,Stockdon = η + SetupStockdon + SwashStockdon

The project formulas sheet also shows an angle-reduction factor, but the current operational code path does not actively apply it.

Source: Stockdon et al. (2006); project formulas sheet; current implementation.

Park and Cox (2016) berm and dune adjustment

Cμ = 0.8 - 0.4 tanh(2Wb / Lo)
Cσ = 1.0 - 0.5 tanh(2Wb / Lo)
βf = arctan(β)
βd = arctan[(zcrest - ztoe) / (0.5Wd)]
α = min(1, max(0, ((η - ztoe) + 0.7Ho) / (1.4Ho)))
βt = (1 - α)βf + αβd
ξ = tan(βuse) / √(Ho/Lo)
R2,Park,μ = η + 1.35 Cμ Ho ξ0.65
R2,Park,σ = η + 1.35 Cσ Ho ξ0.65

The code uses a piecewise choice for βuse: foreshore angle for low water levels, dune/structure face angle for high water levels, and transition angle in between.

Source: Park and Cox (2016); project formulas sheet; current implementation.

Modified Stockdon with Park-style geometry terms

Setupmod,μ = 1.1 Cμ 0.35 tan(βuse) √(LoHo)
Swashmod,μ = 1.1 0.5 Cμ √[HoLo(0.563 tanh2(βuse) + 0.0004)]
R2,mod,μ = η + Setupmod,μ + Swashmod,μ
R2,mod,σ uses Cσ in the same two terms

This hybrid preserves the Stockdon structure while introducing the berm and dune/structure geometry sensitivity from Park and Cox.

Source: project formulas sheet; current implementation.

Oblique-wave reduction factor

γθ = cos(angd)0.4 + 2.7 tan(β)

This factor appears in the formulas sheet and is attributed there to Ormondt et al. (2021). It is useful conceptually, but it is currently inactive in the main Python and browser timestep loop because the live operational runup calculation is using the non-directional baseline noted above.

Source: project formulas sheet; Ormondt et al. (2021) as cited there.

4. Summary metrics used in the Explorer

Expected, conservative, and upper-bound runup

Rmean = mean(R2,Stockdon, R2,Park,μ, R2,mod,μ)
sR = stdev(R2,Stockdon, R2,Park,μ, R2,Park,σ, R2,mod,μ, R2,mod,σ)
ΔH = (Htotal - Hswell) / 2
Expected = Rmean
Conservative = Rmean + sR
Upper Bound = Rmean + sR + ΔH

The upper-bound term uses the difference between total wave height and swell wave height as a simple allowance for local wind-wave or chop uncertainty.

Source: project formulas sheet; current implementation.

Overtopping category logic

If Expected > zcrest : Significant overtopping probable
Else if Conservative > zcrest : Moderate overtopping possible
Else if Upper Bound > zcrest : Minor overtopping possible
Else : Overtopping not expected

This is the decision logic that drives the category pill in the Runup Output panel.

Source: current Explorer implementation in web/app.js.

How to Use the Explorer

  1. Zoom the Selection Map until local transects appear, then click the transect of interest.
  2. Use the top-row toggles in the Geometry Editor to choose dune vs structure geometry, Metric vs English display units, and UTC vs Local Time display.
  3. Edit the toe elevation, toe distance, crest elevation, crest distance, sea-level anomaly, and local waves/chop adjustment as needed.
  4. Select a transect from the map to automatically download live MOP forecast data and the nearest NOAA tide prediction, then populate the lower playback panels.
  5. Edit the beach or structure geometry inputs and the runup and overtopping analysis will refresh automatically.
  6. Use the slider or Play button to move through forecast time and inspect how waves, tides, and runup vary through the event window.

How to Read the Output

  • MOP Wave Forecast: blue line is significant wave height Hs, green dashed line is peak period Tp.
  • NOAA Tide Prediction: tide level in MLLW, with the user-entered sea-level anomaly incorporated into the runup metrics.
  • Playback metrics: Wave Hs, Peak Period, Tide + Anomaly, Max Crest, Expected 2%, Conservative, and Upper Bound all correspond to the selected slider time.
  • Runup profile: shows still water, maximum crest, expected runup, conservative runup, and upper-bound runup projected across the edited profile.
  • Category pill: summarizes whether overtopping is not expected, minor, moderate, or significant, based on comparison with the crest elevation.

Current Implementation Notes

  • The Explorer is fully browser-side at runtime. Live MOP forecasts are pulled directly from CDIP THREDDS and NOAA tides are pulled directly from NOAA CO-OPS.
  • The nearest NOAA tide station is resolved from the California tide-station list included with the web app.
  • The current live Python and browser code path uses the non-directional baseline during the runup calculation loop, even though the formulas sheet includes directional wave-angle terms and an oblique-wave reduction factor.
  • The sea-level anomaly and local waves/chop terms are user inputs, not external forecast products in the present Explorer.
  • The profile is stylized and parameterized. It is not a direct survey profile unless those parameters have been tuned to represent local conditions.
  • If no live MOP forecast is available for a transect, the Explorer falls back to a tide-only preview mode.
