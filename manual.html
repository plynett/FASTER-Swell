Technical Information

Users Manual and Technical Basis

FASTER-Swell is a low-computation coastal forecast workflow for estimating maximum wave crest elevation, 2% exceedance runup, overtopping likelihood, and wave-driven flooding potential along California shorelines. The Explorer combines nearshore wave forecasts from the CDIP MOP alongshore model, NOAA tide predictions, and a simplified user-edited beach or structure profile to produce a transect-by-transect operational view.

Primary objective Provide a short-term swell impact forecast that helps interpret flooding, overtopping, berm performance, and pier or wharf exposure.
Hydrodynamic inputs CDIP MOP wave height, peak period, prediction depth, nearest NOAA tide predictions and recent observed water levels in NAVD88, plus wind-speed data for local chop.
Geometry inputs User-defined slope, toe elevation, toe distance, and crest geometry, plus an auto-estimated sea-level anomaly that remains editable.
Outputs Wave, tide, wind, and chop playback; maximum crest; expected runup; conservative runup; upper-bound runup; and overtopping category.

Purpose and Scope

The background documents describe FASTER-Swell as a forecast tool for large wind-wave and swell events, not a tsunami model. The method is intended for event windows on the order of several days and is designed to answer questions such as: How high can wave crests reach near a pier deck? Will a temporary berm or dune likely be overtopped? Is wave-driven flooding likely on a coastal road or backshore area?

The operational concept is to start with wave conditions already predicted by the MOP system near the 10 m depth contour, combine those conditions with NOAA tide predictions, a recent observed-minus-predicted sea-level anomaly, a wind-derived local chop proxy, and a user-defined shoreline geometry, then evaluate several empirical runup formulations to produce decision-ready water-level estimates.

Operational Data Flow

  1. The Explorer loads statewide MOP transects from the compact data/transects.json web export.
  2. When a transect is selected, the app requests the corresponding CDIP MOP forecast directly from the THREDDS DAP2 ASCII endpoint. If the URL includes start_date=MMDDYYYY, the app instead requests an up-to-five-day CDIP MOP historical window from the matching nowcast or hindcast file. This URL mode is hindcast-only; if the requested five-day window would extend into the present, the end time is clipped to a safe historical hour before now so upstream nowcast files have time to land.
  3. The app resolves the nearest NOAA tide station from the California station list, downloads hourly tide predictions in NAVD88, and estimates observed-minus-predicted sea-level anomaly from NOAA water levels. Forecast mode uses the recent residual window; hindcast mode uses the selected historical window.
  4. In forecast mode, the app requests NWS gridpoint wind-speed forecasts at the transect prediction point and converts those speeds into a local chop-height time series. In hindcast mode, the app requests hourly 10 m historical/reanalysis wind speed from Open-Meteo at the transect prediction point and runs the same chop calculation.
  5. The Geometry Editor defines a stylized profile using the user inputs. Dune mode returns landward to the toe elevation; structure mode remains elevated landward of crest.
  6. For each forecast time, the app computes equivalent deep-water wave properties, breaker conditions, runup estimates, and summary water levels.
  7. The Explorer displays the wave, tide, wind, and chop time series, allows playback through forecast time, and plots the selected-time water levels across the edited profile.

Variable Definitions and Data Sources

Units and datums

The browser model computes internally in metric units. Elevations and water levels are meters relative to NAVD88 unless noted otherwise, distances are meters, wave periods are seconds, wind speed is converted to meters per second for the chop calculation, and angles are radians inside the equations unless a degree-valued source field is explicitly noted.

External forecast and observation variables

Live MOP variables come from the CDIP THREDDS DAP2 ASCII endpoint. If that request is unavailable and a bundled fallback exists, the same fields are read from data/forecasts.

Hswell, Hs
Nearshore swell significant wave height, from CDIP MOP waveHs at the selected transect forecast file.
Tp
Peak wave period, from CDIP MOP waveTp.
d
Prediction-point water depth, from CDIP MOP metaWaterDepth.
Dp
Peak wave direction, from CDIP MOP waveDp. It is loaded and displayed in data structures, but the current runup calculation sets the operational wave-angle terms to zero.
θshore-normal
Transect shore-normal direction in degrees, from CDIP MOP metaShoreNormal when present, otherwise shoreNormal in data/transects.json. It is not active in the current non-directional runup baseline.
ηNOAA,pred, ηpred
NOAA CO-OPS tide prediction in NAVD88 at the nearest tide station, requested from the CO-OPS API with datum=NAVD. Hourly predictions are used for forecast playback; six-minute predictions are used for the recent sea-level anomaly residual calculation.
ηobs
Observed NOAA CO-OPS water level in NAVD88 from the water_level product at the nearest tide station, requested from the CO-OPS API with datum=NAVD and used only for the recent observed-minus-predicted residual estimate.
U10
10 m wind speed at the transect prediction point. Forecast mode loads NWS gridpoint forecast wind speed from forecastGridData.properties.windSpeed. Hindcast mode loads Open-Meteo hourly historical/reanalysis wind_speed_10m. Both sources are converted to meters per second for Hchop.
prediction latitude and longitude
Transect forecast point from data/transects.json, used to request the NWS gridpoint forecast, Open-Meteo historical/reanalysis wind, and resolve the selected map location.
xpier, zpier
Optional pier deck profile points from data/pier_transects/{transect}.csv when available. The CSV uses x as horizontal distance offshore from the shoreline and z as deck elevation in NAVD88.

User-edited geometry and anomaly variables

These variables are editable in the Geometry Editor. Initial geometry values are seeded first from data/known_transects.csv when the selected transect has surveyed beach-profile values, then from data/forecast-manifest.json where transect-specific defaults exist, otherwise from the local generic defaults.

β
Average beach-face slope over the swash zone from the Geometry Editor averageBeachSlope input. When available, this is loaded from the Face Slope column in data/known_transects.csv. Otherwise, the field is auto-populated as twice the MOP average slope, where the MOP average slope is prediction depth divided by backbeach-to-prediction-point distance, then remains editable.
ztoe
Dune toe, berm, or structure base elevation from the beachElevationBase input. When available, this is loaded from the Toe Elevation column in data/known_transects.csv.
Wtoe, Wb
Flat toe or berm width from the bermWidth input. When available, this is loaded from the Toe Distance column in data/known_transects.csv. The same value is used as Park and Cox berm width Wb.
zcrest
Dune, berm, or structure crest elevation from the duneCrestElevation input. When available, this is loaded from the Crest Elevation column in data/known_transects.csv.
Wtoe-crest
Horizontal distance from toe/base to crest from the toeToCrestDistance input. When available, this is loaded from the Crest Distance column in data/known_transects.csv.
profile mode
User toggle that selects dune geometry, which descends landward to ztoe, or structure geometry, which remains at zcrest landward of the crest.
ASL
Sea-level anomaly. It is auto-estimated from NOAA residuals when possible and remains editable through the tideSurgeLevelInput field.

Derived geometry and hydrodynamic variables

zmin
Internal profile plotting reference elevation, fixed at -5 m.
x, xlinear,end, xflat, xc
Cross-shore coordinates generated by the browser from the edited profile geometry.
Wd
Modeled dune or structure width, computed as 2Wtoe-crest.
A, σ, z(x)
Gaussian profile amplitude, width parameter, and resulting profile elevation, derived from ztoe, zcrest, Wd, and xc.
η(t), η
Still-water level used by the runup formulas, computed as NOAA tide prediction plus ASL at the selected playback time.
Hchop, Htotal
Local short-period chop height derived from the active wind-speed source, and total wave-height proxy Hswell + Hchop.
Ls, Lo, k, ns
Shallow-water wavelength, deep-water wavelength, wavenumber, and group factor computed from Tp, d, and g.
angs, angd, Ks, Kr, Ho
Operational wave-angle terms, shoaling coefficient, refraction coefficient, and equivalent deep-water wave height. The current code sets angs = 0, angd = 0, and Kr = 1.
a, b, Ks,b, Kr,b, Hb, dbreaking
Breaking-wave coefficients, breaking-depth shoaling/refraction terms, breaking wave height, and breaking depth solved from the CEM-style breaking relation.
Setup, Swash, R2
Formula-specific setup, swash, and 2% exceedance runup water levels computed from the Stockdon, Park and Cox, and modified Stockdon expressions.

Coefficients, selection terms, and summary variables

g
Gravitational acceleration, fixed at 9.81 m/s2.
Tc
Upper-bound chop period cutoff, fixed at 4 s for the Pierson-Moskowitz short-period proxy.
0.241, 0.13, 1.25
Hchop coefficients from Chop Height Proxy Model.pdf, used directly in the full equation.
MaxRatio, CrestRatio
Maximum-considered wave ratio 1.3 and FEMA crest fraction 0.55 from the project formulas sheet.
Cμ, Cσ
Park and Cox berm-width adjustment coefficients for mean-like and sigma-like runup estimates.
βf, βd, α, βt, βuse
Foreshore angle, dune/structure face angle, transition weight, blended transition angle, and the piecewise angle selected by the current tide level relative to ztoe.
H*, L*, ξ
Numerically floored Ho and Lo values, and the Iribarren-style term used by the Park and modified Stockdon calculations.
γθ
Oblique-wave reduction factor from the formulas sheet. It is documented for completeness but inactive in the current non-directional runup path.
Rmean, sR, FLP, ΔH
Mean of the three central runup estimates, sample standard deviation of the five runup estimates, long period swell uncertainty factor, and local chop uncertainty allowance Hchop/2.
Expected, Conservative, Upper Bound, zcrest,max
Displayed summary water levels at the selected playback time, all computed by the browser from the variables above.

Geometry Representation

Beach face and toe location

zmin = -5 m (internal reference)
xlinear,end = (ztoe - zmin) / β
xflat = xlinear,end + Wtoe

The Explorer builds the seaward profile from a constant average slope β up to the dune or structure toe elevation, then applies a flat toe/berm segment.

Source: current Explorer implementation in app.js.

Dune or structure crest shape

Wd = 2 Wtoe-crest
xc = xflat + Wd / 2
A = zcrest - ztoe
σ = Wd / 6
z(x) = ztoe + A exp(-(x - xc)2 / (2σ2))

The dune/structure crest is represented with a Gaussian rise over the specified toe-to-crest distance. In dune mode the landward side returns to the toe elevation; in structure mode the landward side remains at crest elevation.

Source: current Explorer implementation in app.js.

Hydrodynamic and Runup Equations

Terminology note

The local project files sometimes refer to the Stockdon formulation as "Stockton." In this manual the published name Stockdon et al. (2006) is used. The formulas below match the project formula sheet and the current code path.

1. Water-level anomaly and local-wind preprocessing

Sea-level anomaly from NOAA residuals

ASL = mean(ηobs - ηpred)
η(t) = ηNOAA,pred(t) + ASL

The Explorer uses the nearest NOAA tide station, NAVD88 datum, and a three-day lookback ending one hour before the run. Observed water levels are paired to six-minute predictions in the same datum, with a minimum of 24 paired residuals. If the residual estimate is unavailable, the current sea-level anomaly field value is retained.

Source: current Explorer implementation in app.js.

Local chop proxy from wind speed

Hchop(U10; Tc) = 0.241 (U102 / g) √{1 - exp[-1.25 ((0.13 g Tc) / U10)4]}
Tc = 4 s
Htotal = Hswell + Hchop

The Explorer converts the active wind-speed source to U10 in m/s, evaluates this short-period Pierson-Moskowitz tail proxy at each playback time, and plots the resulting Hchop series on the wind panel. Forecast mode uses NWS gridpoint forecast winds. Hindcast mode uses Open-Meteo hourly historical/reanalysis winds. Non-positive or unavailable wind speeds return Hchop = 0 for that time.

Source: Chop Height Proxy Model.pdf; current implementation in app.js.

2. Conversion from nearshore prediction point to equivalent deep-water properties

Shallow-water and deep-water wavelengths

Ls = (g Tp2 / 2π) √tanh(4π2d / (Tp2 g))
Lo = g Tp2 / (2π)

These are the nearshore and equivalent deep-water wavelengths used throughout the transformation and runup calculations.

Source: project formulas sheet; implemented in FASTER_MOPS.py and app.js.

Group factor, shoaling, and operational deep-water wave height

k = 2π / Ls
ns = 0.5 [1 + 2kd / sinh(2kd)]
angs = 0, angd = 0 (operational baseline)
Ks = √[nsLs / (0.5Lo)]
Kr = 1
Ho = Ks Kr Hs,swell

The documented directional method uses wave direction relative to shoreline normal, but the current Python lineage and Explorer code set the operational directional term to zero inside the timestep loop. MOP wave direction and shore-normal metadata are loaded, but the live runup calculation behaves as this non-directional baseline.

Sources: background docx, formulas sheet, O'Reilly et al. (2016), current code.

3. Breaking-wave calculation and maximum crest elevation

CEM-style breaking relation

a = 43.75 [1 - exp(-19β)]
b = 1.56 / [1 + exp(-19.5β)]
Hb = Ho Ks,b Kr,b
dbreaking = Hb / [b - a Hb / Tp2]

The solver iterates on breaking depth until the Coastal Engineering Manual style breaking relation is satisfied. If no stable bracket is found, the code falls back to Hb = Hswell and dbreaking = Hswell.

Source: project formulas sheet, attributed there to the Coastal Engineering Manual breaking-wave-height method; implemented in Python and web code.

Maximum-considered wave crest

zcrest,max = η + Hb MaxRatio CrestRatio
zsplash,max = η + 2Hb MaxRatio CrestRatio
MaxRatio = 1.3
CrestRatio = 0.55

Here η is NOAA tide prediction plus the current sea-level anomaly. The formula sheet treats 1.3 as a representative ratio between maximum-considered wave height and significant wave height, and 0.55 as the FEMA crest fraction of total wave height. Where a matching pier transect file exists, the runup profile also plots zsplash,max as a thin gray dashed reference line.

Source: project formulas sheet; implemented in FASTER_MOPS.py and app.js.

4. 2% exceedance runup formulas

Stockdon et al. (2006) total runup

SetupStockdon = 1.1 0.35 β √(LoHo)
SwashStockdon = 1.1 0.5 √[HoLo(0.563β2 + 0.0004)]
R2,Stockdon = η + SetupStockdon + SwashStockdon

The project formulas sheet also shows an angle-reduction factor, but the current operational code path does not actively apply it.

Source: Stockdon et al. (2006); project formulas sheet; current implementation.

Park and Cox (2016) berm and dune adjustment

Cμ = 0.8 - 0.4 tanh(2Wb / Lo)
Cσ = 1.0 - 0.5 tanh(2Wb / Lo)
βf = arctan(β)
βd = arctan[(zcrest - ztoe) / (0.5Wd)]
H* = max(Ho, 0.05 m), L* = max(Lo, 0.05 m)
α = min(1, max(0, ((η - ztoe) + 0.7H*) / (1.4H*)))
βt = (1 - α)βf + αβd
βuse = βf if (η - ztoe) / H* < -0.7; βd if > 0.7; otherwise βt
ξ = tan(βuse) / √(H*/L*)
R2,Park,μ = η + 1.35 Cμ H* ξ0.65
R2,Park,σ = η + 1.35 Cσ H* ξ0.65

The code uses a piecewise choice for βuse: foreshore angle for low water levels, dune/structure face angle for high water levels, and transition angle in between.

Source: Park and Cox (2016); project formulas sheet; current implementation.

Modified Stockdon with Park-style geometry terms

Setupmod,μ = 1.1 Cμ 0.35 tan(βuse) √(L*H*)
Swashmod,μ = 1.1 0.5 Cμ √[H*L*(0.563 tanh2(βuse) + 0.0004)]
R2,mod,μ = η + Setupmod,μ + Swashmod,μ
R2,mod,σ uses Cσ in the same two terms

This hybrid preserves the Stockdon structure while introducing the berm and dune/structure geometry sensitivity from Park and Cox.

Source: project formulas sheet; current implementation.

Oblique-wave reduction factor

γθ = cos(angd)0.4 + 2.7 tan(β)

This factor appears in the formulas sheet and is attributed there to Ormondt et al. (2021). It is useful conceptually, but it is currently inactive in the main Python and browser timestep loop because the live operational runup calculation is using the non-directional baseline noted above.

Source: project formulas sheet; Ormondt et al. (2021) as cited there.

5. Summary metrics used in the Explorer

Expected, conservative, and upper-bound runup

Rmean = mean(R2,Stockdon, R2,Park,μ, R2,mod,μ)
sR = stdev(R2,Stockdon, R2,Park,μ, R2,Park,σ, R2,mod,μ, R2,mod,σ)
FLP = (Tp / 18 s)4
ΔH = (Htotal - Hswell) / 2 = Hchop / 2
Expected = Rmean
Conservative = Rmean + FLPsR
Upper Bound = Conservative + ΔH

The long period swell uncertainty factor increases to the fourth power with peak period relative to an 18 s reference period. It is applied to the conservative runup spread to account for potential model error in long-period swell. The upper-bound term then adds the difference between total wave height and swell wave height as a simple allowance for local wind-wave or chop uncertainty. In the Explorer, this local chop height is estimated from the active wind-speed source at each playback time.

Source: project formulas sheet; current implementation.

Overtopping category logic

If Expected > zcrest : Significant overtopping probable
Else if Conservative > zcrest : Moderate overtopping possible
Else if Upper Bound > zcrest : Minor overtopping possible
Else : Overtopping not expected

This is the decision logic that drives the category pill in the Runup Output panel.

Source: current Explorer implementation in app.js.

How to Use the Explorer

  1. Zoom the Selection Map until local transects appear, then click the transect of interest.
  2. Use the top-row toggles in the Geometry Editor to choose dune vs structure geometry, Metric vs English display units, and UTC vs Local Time display. The Explorer opens in English units and Local Time by default.
  3. Review the auto-estimated average beach-face slope and sea-level anomaly, and edit the profile elevations and distances as needed.
  4. Select a transect from the map to automatically download live MOP forecast data and the nearest NOAA tide prediction, then populate the lower playback panels. To run a historical window of up to five days, add start_date=MMDDYYYY to the URL, for example ?transect=OC400&start_date=08212022. Recent start dates are clipped before the current time rather than blended with forecast data.
  5. Edit the beach or structure geometry inputs and the runup and overtopping analysis will refresh automatically.
  6. Use the slider or Play button to move through forecast time and inspect how waves, tides, and runup vary through the event window.

How to Read the Output

  • MOP Wave Forecast: blue line is significant wave height Hs, red dashed line is peak period Tp.
  • NOAA Tide Prediction: tide level in NAVD88, with the current sea-level anomaly incorporated into the runup metrics.
  • NWS Wind Forecast: gold line is wind speed and navy dashed line is the local chop-height proxy estimated from that wind speed.
  • Playback metrics: Wave Hs, Local Chop, Peak Period, Tide + Anomaly, Max Crest, Expected 2%, Conservative, and Upper Bound all correspond to the selected slider time.
  • Runup profile: shows still water, maximum crest, expected runup, conservative runup, and upper-bound runup projected across the edited profile. Where a matching pier file exists, a thick dark-brown line shows the pier deck elevation profile and a thin gray dashed line shows maximum wave splash-up.
  • Category pill: summarizes whether overtopping is not expected, minor, moderate, or significant, based on comparison with the crest elevation.

Current Implementation Notes

  • The Explorer is fully browser-side at runtime. Live MOP forecasts and historical MOP nowcast/hindcast windows are pulled directly from CDIP THREDDS, NOAA tides and water levels are pulled directly from NOAA CO-OPS, current wind forecasts are pulled directly from the NWS API, and hindcast wind speeds are pulled from the Open-Meteo historical weather API.
  • The nearest NOAA tide station is resolved from the California tide-station list included with the web app.
  • The average beach-face slope is auto-estimated from MOP transect geometry as two times the MOP average slope: 2 x prediction depth divided by backbeach-to-prediction-point distance. The value is then limited to the model slope range. This is a swash-zone proxy, not a replacement for surveyed foreshore profile data.
  • The current live Python and browser code path uses the non-directional baseline during the runup calculation loop, even though the formulas sheet includes directional wave-angle terms and an oblique-wave reduction factor.
  • The sea-level anomaly is auto-estimated from NOAA observed-minus-predicted water levels. Forecast mode uses the recent residual window; hindcast mode uses the selected historical playback window.
  • Local chop is auto-estimated from wind speed using the four-second Pierson-Moskowitz short-period proxy documented in the project notes. If no wind interval covers a playback time, the displayed and modeled local chop value for that time is zero.
  • The profile is stylized and parameterized. It is not a direct survey profile unless those parameters have been tuned to represent local conditions.
  • If no live MOP forecast is available for a transect, the Explorer falls back to a tide-only preview mode.
